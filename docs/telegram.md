# Telegram Setup Guide

Noctis collects from Telegram channels using the MTProto protocol via
[gotd/td](https://github.com/gotd/td). This means it connects as a real
Telegram client — not a bot — and receives channel updates in real time.

---

## 1. Create a Telegram API App

Before anything else, you need an `api_id` and `api_hash` from Telegram.

1. Go to [my.telegram.org](https://my.telegram.org) and sign in with your
   Telegram account (use a dedicated/burner account — see [OPSEC](#8-opsec)).
2. Click **API development tools**.
3. Fill in the form:
   - **App title**: anything (e.g. `noctis-collector`)
   - **Short name**: lowercase, no spaces
   - **Platform**: select **Desktop**
4. Submit. Telegram shows your `api_id` (integer) and `api_hash` (string).
   Store them securely — treat the hash like a password.

---

## 2. Configuration

Add the Telegram block to your config file under `noctis.sources.telegram`:

```yaml
noctis:
  sources:
    telegram:
      enabled: true
      apiId: 12345678          # integer from my.telegram.org
      apiHash: "abc123..."     # string from my.telegram.org
      phone: "+15551234567"    # account phone number (for code-based auth)
      password: ""             # 2FA cloud password, if enabled on the account
      sessionFile: /data/telegram.session
      catchupMessages: 50      # fetch last N messages per channel on startup
      channels:
        - username: durov           # public channel by @username
        - id: -1001234567890        # private/joined channel by numeric ID
```

Environment variable substitution is supported: `apiHash: "${TG_API_HASH}"`.

### Config field reference

| Field | Type | Description |
|---|---|---|
| `enabled` | bool | Set to `true` to activate the collector |
| `apiId` | int | API ID from my.telegram.org |
| `apiHash` | string | API hash from my.telegram.org |
| `phone` | string | Account phone number (required for code auth; optional for QR) |
| `password` | string | 2FA cloud password, if enabled on the account |
| `sessionFile` | string | Path where the session file is stored |
| `catchupMessages` | int | Messages to fetch per channel on startup (0 = skip) |
| `channels` | list | Channels to monitor — see [Channel Management](#4-channel-management) |

## 3. Authentication

Noctis needs a valid session before it can collect. There are two paths:
in-cluster QR auth (no shell required) and the standalone `telegram-auth` CLI.

### 3a. QR Auth — In-Cluster (recommended for Kubernetes)

This is the default flow when Noctis starts without an existing session. The
collector initiates QR login and exposes the QR code via the health server.

**Steps:**

1. Deploy Noctis with Telegram enabled in your ConfigMap.

2. Watch logs until the auth URL appears:
   ```
   kubectl logs -f deployment/noctis -n noctis
   ```
   You will see lines like:
   ```
   telegram: starting QR auth — scan the login URL with your Telegram app
   telegram: open Telegram → Settings → Devices → Link Desktop Device
   telegram: scan this URL on your phone  url=tg://login?token=...  expires_in=30s
   ```

3. Port-forward the health server:
   ```
   kubectl port-forward deployment/noctis -n noctis 8080:8080
   ```

4. Open `http://localhost:8080/auth/qr` in a browser.

   The page shows the QR code as a PNG image (generated by go-qrcode, embedded
   as a base64 data URI). It **auto-refreshes every 10 seconds**, so you do not
   need to reload manually when a token expires.

   Alternatively, copy the `tg://login?token=...` URL from the logs and open it
   directly on your phone.

5. In the Telegram app on your phone, go to:
   **Settings → Devices → Link Desktop Device**, then scan the QR code.

6. If 2FA is enabled on the account, Noctis reads the password from the
   `password` config field. If the field is empty, the collector logs an error
   and stops:
   ```
   telegram: 2FA password required but not configured — set 'password' in telegram config
   ```

7. On success, the browser page shows a green checkmark, and the session is
   written to `sessionFile`. Noctis logs:
   ```
   telegram: QR auth successful, session saved
   ```

**QR token lifetime:** tokens expire in approximately 30 seconds. The
`/auth/qr` page fetches a new token automatically every 10 seconds, so
scanning from the browser page is easier than using the raw log URL.

**Idle state:** if the session is already valid when Noctis starts, the
`/auth/qr` page shows "No QR auth pending" and auto-refreshes every 5 seconds.

---

### 3b. CLI Auth — Standalone (`telegram-auth`)

Use this when you want to authenticate outside the cluster and copy the session
file in, or when testing locally.

**QR code in terminal (recommended):**
```
noctis telegram-auth --config config.yaml --qr
```
Renders the QR code directly in the terminal using the qrterminal library.
Phone number is not required for this method. On success, prints:
```
Authentication successful, session saved to /data/telegram.session
```

**Code-based auth (in-app delivery):**
```
noctis telegram-auth --config config.yaml
```
Sends a login code to your Telegram app. Prompts interactively for the code.
Requires `phone` in config.

**Force SMS delivery:**
```
noctis telegram-auth --config config.yaml --sms
```
Same as code auth but requests an SMS resend after the initial in-app delivery.

> `--qr` and `--sms` are mutually exclusive.

**2FA:** if the account has cloud password enabled, the CLI reads it from the
`password` config field. If the field is empty, it prompts interactively:
```
Enter 2FA password:
```

**Session cleanup:** the command removes any stale session file before
attempting auth, ensuring a clean start.

## 4. Channel Management

Channels are configured under `sources.telegram.channels`. Each entry uses
either a `username` or a numeric `id` — not both.

```yaml
channels:
  - username: somePublicChannel   # public channel, resolved by @username
  - id: -1001234567890            # private or already-joined channel by ID
```

**Public channels by username** are resolved via the `ContactsResolveUsername`
Telegram API call, which returns the channel ID and access hash. Noctis
automatically joins public channels via `ChannelsJoinChannel` before
subscribing, so you do not need to join them manually in the app first.

**Private or already-joined channels by ID** use the raw numeric channel ID
with a zero access hash. This works only for channels the account is already
a member of.

Channel names in logs and findings use the username if available, otherwise
`channel:<id>`.

### Auto-Join Public Channels

When a channel is configured by `username`, Noctis automatically calls
`ChannelsJoinChannel` after resolving the username. This means you can add
new public channels to your config and Noctis will join them on startup
without you needing to manually join from the Telegram app.

**Behavior:**

- Only public channels resolved by `username` are auto-joined
- Channels configured by numeric `id` only are assumed already joined
- Private channels and groups are never auto-joined
- If the account is already a member, the join is a no-op (logged as warning)
- Join failures are logged but do not prevent message collection

**OPSEC note:** Auto-joining channels creates a visible membership record.
Your Telegram account will appear in the channel's member list. Use a
dedicated/burner account (see [OPSEC](#8-opsec)).

## 4.1. Runtime Channel Management

Channels can be added while Noctis is running — no restart required.

**Add a channel at runtime:**

```
noctis source add --type telegram_channel --identifier "channelname" -c config.yaml
```

This writes a new entry to the sources database. The Telegram collector polls
the database every 5 minutes for newly added or approved channels and
subscribes to them automatically.

**How config channels and database channels interact:**

- Config channels (`sources.telegram.channels`) are the initial seed. They are
  loaded on startup and merged with approved/active sources from the database.
- All runtime additions go through the database via `source add` or
  `source approve`.
- There is no overlap conflict: the collector deduplicates channels by ID
  before subscribing.

**Approving auto-discovered channels:**

Channels discovered passively (from `t.me/` links or forwarded message
metadata) appear in the sources table with status `discovered`. Approve them
with:

```
noctis source approve <id>
```

Approved channels are picked up within 5 minutes by the next poll cycle.

**Summary of timing:**

| Action | When it takes effect |
|--------|----------------------|
| `source add` | Within 5 minutes (next poll cycle) |
| `source approve` | Within 5 minutes (next poll cycle) |
| Config channel added, pod restarted | Immediately on startup |

## 5. Catchup

Setting `catchupMessages: N` causes Noctis to fetch the last N messages from
each configured channel on startup, before entering the real-time update loop.

Catchup uses the `MessagesGetHistory` Telegram API call with `Limit: N`.

All messages — both catchup and real-time — are deduplicated by a SHA-256
hash of the message content. If the same text appears twice (e.g., a message
caught in catchup and then delivered again as a live update), only the first
instance is processed.

Set to `0` to skip catchup entirely.

---

## 6. Auto-Discovery

Noctis can discover new Telegram channels passively from collected content:

- **Forwarded messages** carry source channel metadata. The collector records
  the originating channel name in the finding's `forward_from` metadata field.
- The discovery engine scans all collected content for `t.me/` links.
- Newly discovered channels appear as `discovered` entries in the sources
  table with source type `telegram`.

To approve a discovered source and begin collecting from it:
```
noctis source approve <id>
```

Auto-approval can be enabled globally via `discovery.autoApprove: true` in
config, but this is not recommended for Telegram due to the join visibility
concern described in [OPSEC](#8-opsec).

## 7. Session Persistence

The session file at `sessionFile` contains the encrypted MTProto session
credentials. Once written, Noctis reuses it on every startup and skips
re-authentication.

**Default deployment — PVC at `/data`:**

The default manifests provision a PVC mounted at `/data`. The session file
survives pod restarts and redeployments. No QR re-scan is needed after a
normal pod restart or rolling update.

```yaml
volumes:
  - name: data
    persistentVolumeClaim:
      claimName: noctis-data
```

**Custom manifests using emptyDir:**

```yaml
volumes:
  - name: data
    emptyDir: {}
```

If you use `emptyDir`, the `/data` volume is ephemeral. The session file is
lost on every pod restart, and you must scan the QR code again. Replace the
`emptyDir` with a PVC for any persistent deployment.

## 8. OPSEC

- **Use a dedicated or burner Telegram account.** Never use a personal account.
  The session file gives full access to the account.
- **Noctis is read-only.** It never sends messages, creates posts, or modifies
  any channel content.
- **Auto-join is visible.** When Noctis joins a public channel via username
  resolution, the account appears in the channel's member list. Channel admins
  can see new members. Consider pre-joining channels manually from the phone
  app and using numeric IDs to avoid triggering join notifications.
- **The session file contains auth credentials.** Treat it as a secret. Do not
  commit it, do not store it in a ConfigMap, and do not include it in container
  images. Mount it from a Secret or a restricted PVC.
- **Consider a separate Telegram account per deployment** (staging vs.
  production) to isolate exposure.

## 9. Troubleshooting

**QR token expires before I can scan it**
The `/auth/qr` page auto-refreshes every 10 seconds and always shows the
latest token. Use the browser page rather than copying the URL from logs.

**Session lost after pod restart**
The default manifests use a PVC at `/data`, so sessions persist across
restarts. If you are using a custom manifest with `emptyDir`, replace it with
a PVC — see [Session Persistence](#7-session-persistence).

**"telegram 2FA password required but not in config"**
The account has cloud password (2FA) enabled. Set the `password` field in the
telegram config block, or use `telegram-auth --qr` which prompts interactively.

**`SESSION_PASSWORD_NEEDED` in logs**
Same cause as above. Add `password: "your-cloud-password"` to the config.

**Flood wait errors**
Telegram is rate-limiting the account. Noctis backs off automatically. Reduce
`catchupMessages` if this happens frequently at startup.

**Channel resolve fails**
- Check the username spelling (without the leading `@`).
- The channel must be public for username resolution to work.
- If the channel is private, join it manually in the Telegram app first, then
  reference it by numeric ID.

**"resolved peer is not a channel"**
The username resolves to a user or a group, not a broadcast channel. Verify
the channel type in the Telegram app.

**Auth QR page shows "No QR auth pending"**
Either the session is already valid (no auth needed), the collector has not
started yet, or it is not enabled in config. Check logs and confirm
`enabled: true` in the telegram config block.
